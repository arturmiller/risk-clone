# Phase 9: Riverpod Providers and Persistence - Research

**Researched:** 2026-03-15
**Domain:** Riverpod 3.x AsyncNotifier, Flutter app lifecycle, ObjectBox persistence, serial action queue
**Confidence:** HIGH

---

## Summary

Phase 9 wires the state layer that bridges the already-complete engine (Phases 6–8) with the UI layers to come (Phases 10–11). The core deliverable is `GameNotifier` — an `AsyncNotifier<GameState>` that owns canonical game state, processes human moves and bot turns serially, and persists state to ObjectBox on app backgrounding.

The infrastructure is almost entirely in place: ObjectBox `SaveSlot` entity is registered and live in `main.dart`, `mapGraphProvider` already loads the classic map, the `PlayerAgent` interface is stable, and `Isolate.run()` has been confirmed to pass freezed `GameState` and `MapGraph` across boundaries without any serialization workaround. Phase 9's job is to build `GameNotifier`, `UIStateNotifier`, and the lifecycle observer on top of that foundation.

The key technical decisions for this phase are: (1) use `AppLifecycleState.hidden` (not `inactive`) as the save trigger — `hidden` fires reliably on both iOS and Android when all views become invisible, which is the correct semantics for "the user has left the app"; (2) a single `GameNotifier` method queue via `async/await` and Dart's single-threaded event loop is sufficient to prevent race conditions — no explicit lock is needed because isolate callbacks return to the main isolate's event queue serially; (3) `UIStateNotifier` holds selection state only (`Notifier<UIState>`, not `AsyncNotifier`) because it never performs async work.

**Primary recommendation:** Build `GameNotifier` as a `@riverpod`-annotated `AsyncNotifier<GameState>` with explicit `state = const AsyncLoading()` before and `state = await AsyncValue.guard(...)` after any async operation. Register an `AppLifecycleListener` in `GameNotifier`'s `build()` method, disposing it via `ref.onDispose`. Save to ObjectBox on `onHide`. Resume detection happens in `build()` by querying the ObjectBox box for an existing save slot.

### Pattern 1: GameNotifier as AsyncNotifier

**What:** `GameNotifier` extends `AsyncNotifier<GameState>`. `build()` checks ObjectBox for a resume slot and returns either the restored state or an empty/unstarted sentinel. All state mutations (setup, human move, bot turn) go through async methods that set `AsyncLoading` then `AsyncValue.guard`.

**When to use:** Always. This is the single source of truth for game state.

**Example:**
```dart
// Source: Riverpod 3.x official pattern (riverpod.dev/docs/whats_new)
import 'package:riverpod_annotation/riverpod_annotation.dart';

part 'game_provider.g.dart';

@riverpod
class GameNotifier extends _$GameNotifier {
  AppLifecycleListener? _lifecycleListener;

  @override
  Future<GameState?> build() async {
    // Check for an existing save on cold start
    final store = ref.read(storeProvider);
    final box = store.box<SaveSlot>();
    final slots = box.getAll();
    if (slots.isNotEmpty) {
      // Return saved state so UI can offer resume prompt
      final json = jsonDecode(slots.first.gameStateJson) as Map<String, dynamic>;
      final restored = GameState.fromJson(json);
      _registerLifecycleListener();
      return restored;
    }
    _registerLifecycleListener();
    return null; // no game in progress
  }

  void _registerLifecycleListener() {
    _lifecycleListener = AppLifecycleListener(
      onHide: _saveState,
    );
    ref.onDispose(() => _lifecycleListener?.dispose());
  }

  Future<void> _saveState() async {
    final current = state.valueOrNull;
    if (current == null) return;
    final store = ref.read(storeProvider);
    final box = store.box<SaveSlot>();
    final json = jsonEncode(current.toJson());
    final slots = box.getAll();
    final slot = slots.isNotEmpty
        ? (slots.first..gameStateJson = json..turnNumber = current.turnNumber
            ..timestamp = DateTime.now().toIso8601String())
        : SaveSlot(
            gameStateJson: json,
            turnNumber: current.turnNumber,
            timestamp: DateTime.now().toIso8601String(),
          );
    box.put(slot);
  }

  Future<void> setupGame(GameConfig config) async {
    state = const AsyncLoading();
    state = await AsyncValue.guard(() async {
      final mapGraph = await ref.read(mapGraphProvider.future);
      return SetupEngine.setupGame(mapGraph, config.playerCount, rng: Random());
    });
  }

  Future<void> submitHumanAction(PlayerAction action) async {
    final current = state.requireValue!;
    state = const AsyncLoading();
    state = await AsyncValue.guard(() async {
      return _applyAction(current, action);
    });
  }

  Future<void> runBotTurn() async {
    final current = state.requireValue!;
    final mapGraph = await ref.read(mapGraphProvider.future);
    state = const AsyncLoading();
    state = await AsyncValue.guard(() async {
      final agents = _buildAgents(current);
      return await Isolate.run(() {
        final (newState, _) = executeTurn(current, mapGraph, agents, Random());
        return newState;
      });
    });
  }

  Future<void> clearSave() async {
    final store = ref.read(storeProvider);
    store.box<SaveSlot>().removeAll();
    state = const AsyncData(null);
  }
}
```

### Pattern 2: UIStateNotifier as Synchronous Notifier

**What:** `UIStateNotifier` is a plain `Notifier<UIState>` (not `AsyncNotifier`) because selection state is always synchronous. It holds: selected source territory, valid targets, current phase prompt.

**When to use:** For all ephemeral UI state that the engine does not need to know about.

**Example:**
```dart
// Source: Riverpod 3.x official patterns
@riverpod
class UIStateNotifier extends _$UIStateNotifier {
  @override
  UIState build() => const UIState.empty();

  void selectTerritory(String name, GameState gameState, MapGraph mapGraph) {
    // compute valid targets based on current game phase and selected source
    state = state.copyWith(selectedTerritory: name, validTargets: _computeTargets(...));
  }

  void clearSelection() {
    state = const UIState.empty();
  }
}

@freezed
abstract class UIState with _$UIState {
  const factory UIState({
    String? selectedTerritory,
    @Default({}) Set<String> validTargets,
  }) = _UIState;

  factory UIState.empty() => const UIState();
}
```

### Pattern 3: AppLifecycleListener in Provider (Flutter 3.13+)

**What:** `AppLifecycleListener` is the current API (replaces mixin-based `WidgetsBindingObserver`). Created in `build()`, disposed via `ref.onDispose()`. The `onHide` callback is the correct save trigger on both iOS and Android.

**Why `onHide` not `onInactive`:**
- `inactive` = app still visible, just lost input focus (phone call overlay, app switcher entering view). This fires frequently for non-backgrounding events. Saving here is premature.
- `hidden` = all views are invisible. This is the correct semantics for "user has left the app." Fires on iOS and Android before `paused`.
- The SUMMARY.md originally said `inactive` but the Flutter 3.13 lifecycle model with the `hidden` state makes `onHide` the correct choice.

**Example:**
```dart
// Source: Flutter 3.13+ AppLifecycleListener API (api.flutter.dev/flutter/widgets/AppLifecycleListener-class.html)
_lifecycleListener = AppLifecycleListener(
  onHide: () async {
    await _saveState();
  },
);
// In ref.onDispose:
ref.onDispose(() => _lifecycleListener?.dispose());
```

### Pattern 4: ProviderContainer Testing (No Flutter Widget Required)

**What:** `ProviderContainer` from `package:riverpod/riverpod.dart` allows testing `AsyncNotifier` logic without a widget tree. Override providers with fakes using `overrides:` parameter.

**When to use:** All `GameNotifier` and `UIStateNotifier` tests.

**Example:**
```dart
// Source: Riverpod testing patterns (codewithandrea.com/articles/unit-test-async-notifier-riverpod)
test('setupGame transitions to AsyncData', () async {
  final fakeStore = FakeStore();
  final container = ProviderContainer(
    overrides: [
      storeProvider.overrideWithValue(fakeStore),
      mapGraphProvider.overrideWith((ref) async => testMapGraph),
    ],
  );
  addTearDown(container.dispose);

  final notifier = container.read(gameNotifierProvider.notifier);
  await notifier.setupGame(GameConfig(playerCount: 3, difficulty: Difficulty.medium));

  final result = container.read(gameNotifierProvider);
  expect(result, isA<AsyncData<GameState?>>());
  expect(result.requireValue, isNotNull);
});
```

### Pattern 5: Serial Action Queue via Async/Await

**What:** Dart's single-threaded event loop plus `async/await` provides natural serialization. When `GameNotifier.runBotTurn()` is called while a previous call is still executing, the second call's microtask queues behind the first. No explicit mutex is needed.

**Caveat:** If the UI calls two notifier methods in the same synchronous frame (e.g., button tap triggers `submitHumanAction` AND some listener triggers `runBotTurn` immediately), both calls start before either `await` resolves. To make this explicit, add a `_processing` guard:

```dart
bool _processing = false;

Future<void> _withLock(Future<void> Function() fn) async {
  if (_processing) return; // drop duplicate calls
  _processing = true;
  try {
    await fn();
  } finally {
    _processing = false;
  }
}
```

### Anti-Patterns to Avoid

- **Game logic in GameNotifier:** GameNotifier is an orchestrator, not an engine. All combat/card/fortify logic stays in `engine/`. GameNotifier calls engine functions and stores results.
- **Saving on `inactive` instead of `hidden`:** `inactive` fires on phone call overlays and app-switcher-entering events. This causes spurious saves. Use `onHide`.
- **UIState inside GameState:** Selection state (selected territory, valid targets) must never be in `GameState`. It pollutes engine equality and complicates snapshot tests.
- **Reconstructing MapGraph on every notifier call:** MapGraph is static. Read it once from `mapGraphProvider.future` in each notifier method, or cache it in `build()`. Do not rebuild it from JSON on every bot turn.
- **Calling `state.requireValue` without null check when state type is `GameState?`:** Since `build()` returns `GameState?` (null = no active game), all notifier methods must guard `if (current == null) return`.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Async state with loading/error | Manual `isLoading` bool + `error` field | `AsyncValue<T>` from Riverpod | Built-in loading/error/data states; UI can pattern-match |

## Common Pitfalls

### Pitfall 1: Using `inactive` Instead of `hidden` for Save Trigger

**What goes wrong:** `AppLifecycleState.inactive` fires when the user enters the app switcher, gets a phone call overlay, or uses TouchID. The game state is saved spuriously on every such event. On resume, state is correct but the save slot is stale with mid-session data unnecessarily.

**Why it happens:** Legacy tutorials use `inactive` because `hidden` was added in Flutter 3.13. Pre-3.13 code has no `hidden` case.

**How to avoid:** Use `AppLifecycleListener.onHide` callback. This fires when all views are invisible (both iOS and Android), which is the correct "user left the app" semantics.

**Warning signs:** Save slot `timestamp` updates every time the user opens the notification shade.

### Pitfall 2: Forgetting `ref.onDispose` for AppLifecycleListener

**What goes wrong:** `AppLifecycleListener` is not disposed when the provider is rebuilt. Multiple listeners accumulate, each calling `_saveState()` independently. On first app background, the save is called multiple times.

**How to avoid:** Always register disposal in `build()`:
```dart
ref.onDispose(() => _lifecycleListener?.dispose());
```

### Pitfall 3: `state.requireValue` Throws on `GameState?` Type

**What goes wrong:** `GameNotifier` returns `GameState?` (null when no game in progress). Calling `state.requireValue` on a null-value `AsyncData<GameState?>` succeeds but returns null. Subsequent null-dereference throws at runtime.

**How to avoid:** Guard every notifier method: `final current = state.valueOrNull; if (current == null) return;`

### Pitfall 4: Calling `mapGraphProvider.future` From Inside `Isolate.run`

**What goes wrong:** `ref` is not available inside `Isolate.run` closures. Attempting to call `ref.read(mapGraphProvider.future)` inside the isolate closure causes a "ref used after disposal" error or a null dereference.

**How to avoid:** Read the map graph before entering the isolate:
```dart
Future<void> runBotTurn() async {
  final current = state.requireValue!;
  final mapGraph = await ref.read(mapGraphProvider.future); // ← BEFORE Isolate.run
  state = const AsyncLoading();
  state = await AsyncValue.guard(() async {
    return await Isolate.run(() {
      final (newState, _) = executeTurn(current, mapGraph, agents, Random());
      return newState;
    });
  });
}
```

### Pitfall 5: ObjectBox `box.put` During `Isolate.run`

**What goes wrong:** `Store` and `Box` are not isolate-safe. Attempting to write to ObjectBox from within `Isolate.run` causes a crash or silent data corruption.

**How to avoid:** All ObjectBox operations happen on the main isolate. `_saveState()` is called from the lifecycle callback (main isolate). Bot turn computation happens in the isolate; only the returned `GameState` value crosses back to the main isolate where ObjectBox writes occur.

### Pitfall 6: `build_runner` Not Re-run After Adding game_provider.dart

**What goes wrong:** `GameNotifier` is annotated with `@riverpod` but `game_provider.g.dart` has not been generated. Imports fail with "Target of URI doesn't exist: 'game_provider.g.dart'".

**How to avoid:** Run `dart run build_runner build --delete-conflicting-outputs` in `mobile/` after creating `game_provider.dart` and `ui_provider.dart`. This is a Wave 0 task.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `WidgetsBindingObserver` mixin | `AppLifecycleListener` class | Flutter 3.13 (stable Aug 2023) | Cleaner API; no mixin boilerplate; dispose is explicit |
| `AppLifecycleState.inactive` save trigger | `AppLifecycleState.hidden` save trigger | Flutter 3.13 added `hidden` state | More precise: fires only when app actually invisible, not on overlays |
| `StateNotifier` | `Notifier` / `AsyncNotifier` | Riverpod 2.0 (2022), standard in 3.0 (2025) | `StateNotifier` is deprecated in Riverpod 3.x |
| Manual provider declaration | `@riverpod` annotation + generator | Riverpod 2.0+ | Eliminates boilerplate, type-safe |

**Deprecated/outdated:**
- `StateNotifier` + `StateNotifierProvider`: Deprecated in Riverpod 3.x. Use `Notifier` / `AsyncNotifier` instead.
- `WidgetsBindingObserver` mixin on `State<T>`: Still works but `AppLifecycleListener` is the modern replacement for Flutter 3.13+.
- `didChangeAppLifecycleState(AppLifecycleState.inactive)` as save point: Superseded by `AppLifecycleListener.onHide` in Flutter 3.13+.

## Open Questions

1. **HumanAgent implementation strategy**
   - What we know: `PlayerAgent` abstract class requires 5 methods. Human input is captured by UI taps, not a single `PlayerAgent` instance.
   - What's unclear: Should `GameNotifier` expose individual methods per phase action (e.g., `placeArmies(String territory, int count)`, `submitAttack(...)`, `submitFortify(...)`), or should it accept a union type `PlayerAction` sealed class?
   - Recommendation: Individual typed methods per phase is cleaner for the UI to call and easier to unit test. Define in Phase 9; UI phases (10–11) will call these methods.

2. **Resume prompt location**
   - What we know: `GameNotifier.build()` returns `GameState?`. If non-null, a game can be resumed.
   - What's unclear: Should the resume prompt live in `HomeScreen` (watches `gameNotifierProvider`, shows "Resume" button if non-null) or as a dedicated `ResumeScreen`?
   - Recommendation: `HomeScreen` watches the provider and conditionally shows a "Resume" or "New Game" button — simpler, no extra route needed, consistent with single-save-slot design.

3. **`GameConfig` model**
   - What we know: `setupGame` needs player count and difficulty. No `GameConfig` model exists yet.
   - What's unclear: Where does it live? engine/models/ (if engine needs it) or providers/ (if only GameNotifier needs it)?
   - Recommendation: Create `mobile/lib/engine/models/game_config.dart` as a plain (non-freezed) class; engine setup function already takes `numPlayers` and `rng`, so config just wraps those for the notifier layer.
