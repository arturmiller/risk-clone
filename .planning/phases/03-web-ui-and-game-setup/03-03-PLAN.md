  <files>risk/static/app.js, risk/static/map.js</files>
  <action>
    Create `risk/static/app.js` -- the main application controller:

    1. **State management:**
       - `let gameState = null;` -- latest GameState from server
       - `let currentInputType = null;` -- what input the server is waiting for
       - `let selectedSource = null;` -- first click in click-click model
       - `let validSources = [];` and `let validTargets = [];`
       - `let ws = null;` -- WebSocket connection
       - `const PLAYER_COLORS = ['#e74c3c', '#3498db', '#2ecc71', '#f1c40f', '#9b59b6', '#e67e22'];`
       - `const PLAYER_NAMES = ['Player 1 (You)', 'Player 2', 'Player 3', 'Player 4', 'Player 5', 'Player 6'];`
       - `const PHASE_NAMES = {1: 'Reinforce', 2: 'Attack', 3: 'Fortify'};` (matching TurnPhase enum values)

    2. **Setup screen handler:**
       - On `#start-btn` click: get player count from select, open WebSocket to `ws://${window.location.host}/ws`, on open send `{"type": "start_game", "num_players": N}`. Hide `#setup-screen`, show `#game-board`. Load SVG map into `#map-container` via fetch(`/static/classic_map.svg`) if not already loaded.

    3. **WebSocket message handler (`ws.onmessage`):**
       - Parse JSON, switch on `msg.type`:
       - `"game_state"`: Store state. Call `updateMap(msg.state)` and `updateSidebar(msg.state, msg.continent_info)`. If msg.prompt, call `showBanner(msg.prompt)` else `hideBanner()`.
       - `"request_input"`: Store `currentInputType = msg.input_type`, `validSources = msg.valid_sources || []`, `validTargets = msg.valid_targets || []`. Call `enableInputMode(msg)` which highlights valid sources on map and shows appropriate UI controls. For reinforcement: show remaining armies counter. For card_trade: show card trade panel with player's cards.
       - `"game_event"`: Call `appendGameLog(msg)`.
       - `"game_over"`: Call `showGameOver(msg)`.

    4. **Action sending:**
       - `sendAction(actionType, data)`: sends `{"type": "player_action", "action_type": actionType, "data": data}` over WebSocket.
       - End Attack Phase button: `sendAction("end_phase", {})`.
       - Skip Fortify button: `sendAction("end_phase", {})`.

    5. **Input modes:**
       - `enableInputMode(msg)`: Based on input_type, configure the click handler behavior:
         - `choose_reinforcement_placement`: Click owned territory -> show number input (prompt or small modal) for how many armies (1 to remaining). On confirm, send reinforce action with placements dict. Track remaining armies, allow multiple placements until all placed.
         - `choose_attack`: Highlight valid sources. First click selects source (highlight it, then request valid targets from state -- compute client-side: neighbors of source owned by enemy). Second click on valid target sends attack action. Show dice selector (default max). Include "End Attack Phase" button.
         - `choose_fortify`: Similar click-click as attack but for friendly connected territories. Include "Skip Fortify" button.
         - `choose_card_trade`: Show card trade panel with cards. Player selects 3 cards, clicks Trade. If forced, must trade.

    6. **Game over:**
       - Show `#game-over` overlay with win/loss message and "New Game" button.
       - "New Game" button: close WebSocket, hide game board, show setup screen, reset state.

    Create `risk/static/map.js` -- SVG map rendering and interaction:

    1. **`loadMap()`**: Fetch `/static/classic_map.svg`, insert into `#map-container` via innerHTML. Then attach click handlers to all `[data-territory]` elements. Return a promise.

    2. **`updateMap(state)`**: For each territory in state.territories:
       - Set fill color: `el.style.fill = PLAYER_COLORS[territory.owner]`
       - Update army label: find `[data-army-label="name"]`, set textContent to territory.armies

    3. **`highlightValidSources(sources)`**: Add `.valid-source` class to territories in sources list. Add `.dimmed` class to all others. Clear any previous highlights first.

    4. **`highlightValidTargets(source, targets)`**: Add `.selected` class to source. Add `.valid-target` class to targets. Dim others.

    5. **`clearHighlights()`**: Remove all `.selected`, `.valid-source`, `.valid-target`, `.dimmed` classes from all territories.

    6. **Territory click handler**: Calls `handleTerritoryClick(territoryName)` which is defined in app.js based on current input mode.

    7. **Client-side target computation**: `getValidAttackTargets(source, state)` -- returns neighbor territory names where owner != source owner. `getValidFortifyTargets(source, state)` -- for v1, just show adjacent friendly territories (full connected-path validation happens server-side). These are used to highlight targets after source selection WITHOUT a round-trip to server.

    Important implementation notes:
    - SVG is served from `/static/classic_map.svg` (the static files mount serves from risk/static/, but the SVG is in risk/data/). The app.py needs to also serve the SVG. Either copy SVG to static/ or add a route. Simplest: load SVG in the fetch from a dedicated route, or copy the SVG to risk/static/ during build. For simplicity, fetch from `/static/classic_map.svg` and ensure Plan 01's app.py serves it (may need an additional route or symlink -- document this for executor). Alternative: just use a direct route `/map` that returns the SVG file.
    - Actually, simplest approach: in `app.js loadMap()`, fetch the SVG from `/api/map` (a dedicated FastAPI route that returns the SVG file). Add this route to app.py: `@app.get("/api/map") async def get_map(): return FileResponse("risk/data/classic_map.svg", media_type="image/svg+xml")`.
    - Reinforcement placement needs special handling: it's multi-step (place armies one territory at a time until all placed, then send the full placements dict). Track placements locally, show remaining counter, send when all placed.
