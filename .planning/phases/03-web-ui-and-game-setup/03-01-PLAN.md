<objective>
Build the server-side infrastructure for the Risk web UI: FastAPI application with WebSocket endpoint, typed message protocol, HumanWebSocketAgent bridging sync game loop to async WebSocket, and GameManager running the game in a background thread.

Purpose: This is the backend foundation that the frontend will communicate with. The message protocol defines the contract between server and client. The HumanWebSocketAgent is the critical async/sync bridge that lets the synchronous execute_turn() loop pause and wait for human input over WebSocket.

Output: Working FastAPI server with WebSocket endpoint, all server-side Python modules, and unit tests for message serialization and agent queue bridging.
</objective>

    1. Create `risk/server/human_agent.py`:
       - `HumanWebSocketAgent` class that implements all 6 PlayerAgent protocol methods.
       - Constructor takes `loop: asyncio.AbstractEventLoop` and stores an `asyncio.Queue` for input.
       - `set_send_callback(callback)` to register a function that sends messages to the browser.
       - `set_map_graph(map_graph)` to store map graph reference for computing valid targets.
       - Each `choose_*` method: (a) computes valid options from state + map_graph, (b) sends a `RequestInputMessage` via callback with valid sources/targets, (c) calls `_wait_for_input()` which blocks using `asyncio.run_coroutine_threadsafe(self._input_queue.get(), self._loop).result()`.
       - `choose_defender_dice` always returns `max_dice` (per user decision: human auto-defends with max dice).
       - `choose_blitz` always returns `None` (human uses regular attacks with dice control).
       - `receive_input(data: dict)` is called from the async WebSocket handler to put data into the queue.
       - For `choose_attack`: valid_sources = territories owned by player with armies >= 2 that have enemy neighbors. valid_targets computed after source selection (sent in separate message).
       - For `choose_reinforcement_placement`: sends armies count, player's owned territories.
       - For `choose_fortify`: valid_sources = owned territories with armies >= 2.
       - For `choose_card_trade`: sends player's cards and whether trade is forced.

    2. Create `risk/server/game_manager.py`:
       - `GameManager` class managing game lifecycle.
       - `start_game(num_players, websocket)`: Creates HumanWebSocketAgent for player 0, RandomAgent for players 1..N-1. Sets up send callback to push messages via websocket. Starts game loop in background thread via `asyncio.get_event_loop().run_in_executor(None, self._run_game_loop, ...)`.
       - `_run_game_loop(agents, map_graph, rng)`: Wraps run_game() but does NOT call run_game() directly. Instead, replicates the setup + loop pattern from run_game() so it can inject delays for bot turns (time.sleep(0.5) between bot actions) and emit game events. Specifically: calls setup_game(), then loops calling execute_turn(), and after each turn sends updated GameState to browser. Between bot turns, sleep 0.5s per user decision. Emits GameEventMessages for key events.
       - `handle_player_action(data)`: Calls `human_agent.receive_input(data["data"])`.
       - `cancel_game()`: Sets a cancellation flag checked between turns.
       - Event emission: After each execute_turn call, compare old state vs new state to detect: attacks (territory owner changed), conquests, eliminations (player.is_alive changed), card trades (trade_count changed). Emit GameEventMessage for each detected event.
       - Note: The game manager needs to wrap execute_turn more granularly to emit per-action events. For v1, emit events at the turn level by diffing state before/after execute_turn. Per-action events would require engine modifications deferred to later.
